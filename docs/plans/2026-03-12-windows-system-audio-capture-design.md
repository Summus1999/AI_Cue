# Windows System Audio Capture Design

## 目标

为 `AI_Cue` 增加 Windows 平台下的系统输出音频录制能力。录制目标限定为默认播放设备上的系统总输出音频，点击开始后开始采集，点击停止后一次性返回 `WAV` 字节数据给前端。当前阶段不实现实时转写，不实现设备选择 UI，但需要为后续实时转写保留清晰的扩展边界。

## 范围

本设计仅覆盖以下能力：

- 仅支持 Windows
- 仅捕获默认播放设备的系统总输出音频
- 仅提供开始录音与停止录音接口
- 停止录音后一次性返回 `WAV` 数据
- 前端保留当前录音按钮交互

本设计不覆盖以下能力：

- 实时流式转写
- 用户手动选择输出设备
- 指定进程音频捕获
- 长时间持久化录音文件管理

## 问题背景

当前工作区中的 `src-tauri/src/audio.rs` 是基于 `cpal` 的简化实现，其核心问题不是单纯的编译错误，而是技术路径本身与目标不一致。它尝试将输出设备与输入流接口拼接使用，这并不等价于 Windows 系统输出音频回采。即便修复局部类型错误，也无法保证得到正确的系统输出音频数据。

因此，本次设计不沿用现有简化实现，而是直接收敛到 Windows 标准能力 `WASAPI loopback`。

## 技术选型

### 采集方案

采用 Windows Core Audio / WASAPI 的 loopback capture 能力，从默认渲染设备抓取系统混音后的 PCM 数据。

理由如下：

- 与“录制电脑输出音频”的目标完全一致
- 是 Windows 上标准、稳定、可解释的实现路径
- 便于后续扩展实时转写
- 避免继续在跨平台抽象层上做错误适配

### Rust 模块拆分

建议将当前单文件实现重构为以下结构：

- `src-tauri/src/audio/mod.rs`
- `src-tauri/src/audio/types.rs`
- `src-tauri/src/audio/recorder.rs`
- `src-tauri/src/audio/windows_wasapi.rs`

其中：

- `mod.rs` 对外暴露统一接口
- `types.rs` 定义录音状态、错误和格式信息
- `recorder.rs` 负责录音状态机、线程控制与会话管理
- `windows_wasapi.rs` 负责 Windows 底层采集实现

这样可以把平台细节与上层命令接口彻底隔离，避免后续 ASR 集成时继续污染底层模块。

## 架构设计

整体调用链如下：

1. 前端点击录音按钮
2. Tauri 调用 `start_audio_recording`
3. 命令层调用录音器 `start`
4. 录音器创建采集会话并启动后台线程
5. 线程内初始化 COM 与 WASAPI 资源
6. 获取默认渲染设备
7. 以 loopback 模式持续读取 PCM 数据
8. 前端点击停止按钮
9. Tauri 调用 `stop_audio_recording`
10. 录音器发送停止信号并等待线程退出
11. 线程返回原始 PCM 缓冲
12. 停止流程统一完成格式转换与 WAV 封装
13. 命令层将 `Vec<u8>` 返回给前端

这里有两个关键边界：

- 前端不感知 WASAPI 细节
- Tauri 命令层不直接管理底层采集资源

## 状态模型

录音器内部采用明确状态机，而不是多个分散的布尔标志：

- `Idle`
- `Starting`
- `Recording`
- `Stopping`
- `Failed`

状态机的意义如下：

- 防止重复开始或重复停止
- 防止线程启动失败时前端误判为正在录音
- 为后续增加状态查询、时长统计、错误回传提供统一承载点

录音器建议维护以下状态：

- 当前状态
- 停止信号
- 后台线程句柄
- 原始音频缓冲区
- 输入音频格式
- 最近一次错误上下文

## 音频格式策略

WASAPI loopback 获取到的通常是设备当前混音格式，而不是项目想要的标准 ASR 输入格式。因此本设计将音频处理分为两层：

- 采集层：按设备原始混音格式稳定采集
- 输出层：在停止录音后统一转换为项目标准格式

项目输出标准格式建议固定为：

- 单声道
- 16kHz
- PCM 16-bit
- WAV 容器

这样设计的原因如下：

- 与常见语音识别服务兼容度高
- 有利于后续阿里云语音识别接入
- 调试时可直接保存与试听
- 避免采集阶段强行要求设备必须支持目标格式

## 线程与生命周期

录音流程需要以会话为单位管理资源，而不是依赖全局共享布尔变量。建议设计为：

- `start` 时创建新的录音会话
- 会话内部拥有后台线程与采集资源
- `stop` 时消费当前会话并等待其完整退出
- 会话结束后再回到 `Idle`

线程模型要求：

- 后台线程负责初始化和释放 COM/WASAPI 资源
- 主线程只做控制与结果收集
- 停止时必须等待线程退出，不能简单丢弃

这样可以避免：

- 资源悬挂
- 多次开始停止后状态污染
- 线程未退出导致的下一次启动失败

## 错误处理

错误处理分为三层：

### 用户可见错误

- 当前没有默认播放设备
- 录音尚未开始
- 录音线程启动失败
- 当前正在录音中，不能重复开始

### 内部技术错误

- COM 初始化失败
- WASAPI 接口获取失败
- 音频客户端初始化失败
- 捕获缓冲读取失败
- 格式转换失败

### 诊断上下文

- 失败阶段
- 设备名称
- HRESULT 或底层错误文本
- 当前状态

对前端返回时应保持错误信息简洁明确，对开发阶段日志则保留足够上下文。

## 日志策略

本项目要求减少无意义日志，因此日志策略如下：

- 默认仅记录关键状态变化
- 不在采集循环中高频打印
- 错误日志必须包含失败阶段与必要上下文
- 不保留调试阶段的噪声式 `println!`

## 前端配合改造

前端改造应尽量保持轻量，重点如下：

- 修复现有 `TypeScript` 严格模式错误，恢复前端可构建状态
- 点击开始后，仅在后端成功返回时进入录音状态
- 点击停止后，等待后端返回 `WAV` 数据
- 失败时恢复按钮状态并提示用户
- 不在本轮引入实时转写逻辑

前端仍然只依赖两个命令：

- `start_audio_recording`
- `stop_audio_recording`

## 扩展点设计

虽然本轮不实现实时转写，但架构上需要为其预留扩展点。预留方式如下：

- 录音器内部使用“原始 PCM 缓冲”作为中间表达
- 将“采集”和“编码/封装”分离
- 后续实时转写时，可以在采集循环中把 PCM 分块推送给转写层

这样后续扩展实时转写时，不需要推倒重写底层捕获模块。

## 验收标准

### 构建验收

- `cargo build` 成功
- `npm run build` 成功
- `tauri dev` 可启动

### 功能验收

- 系统播放音频时，点击开始录音能够正常进入录音状态
- 停止后能够返回非空 `WAV` 数据
- 返回的 `WAV` 可被播放器正常打开
- 音频内容与系统输出基本一致
- 静音场景下不崩溃
- 多次开始/停止后状态仍然正确
- 无默认播放设备时能给出明确错误

## 实施原则

- 不保留当前 `audio.rs` 的错误采集路径
- 优先保证技术路径正确，再追求功能扩展
- 本轮不引入设备选择和实时转写，避免需求扩散
- 模块边界必须为后续 ASR 和实时转写保留扩展能力
